Web Behavior Tracker

Introduction

The web behavior tracker is a library that makes it easy for teams to track and send web behavior events to SpareBank1's data platform.

Documentation

If you are reading this readme in stash, we recommend you head over to our documentation page.

Navigating the documentation page

Here is a general overview of where and what we provide of documentation:

The main page is the Modules/Index page, here you'll have an overview of everything the tracker exposes. You can go to this page by clicking the link "Web Behavior Tracker" in the header or in the sidebar (under modules). For our written guides on general tracker usage and guidelines, read the documents listed under "Modules" in the sidebar to get started. As for tracker functionality, see the WebBehaviorTracker interface to explore the available tracking functions.

On the main page we have categorized the exposed definitions for your convenience, they are as follows (listed in order):

  • WebBehaviorTracker: Functions for interacting with the tracker, plus the interface listed above.
  • InitParams (tracker params): Parameters used when initializing the tracker.
  • Props (event params): Parameters used when tracking events.
  • Enums: Relevant enumerations to have at hand for tracker usage.
  • Literal Union Types: A literal union type is a set of possible values, i.e.: Language: "en" | "nb" | "nn". The values are derived from the enums.
  • Helper functions: Utility functions for simplifying tracker usage.
  • Constants: Easily accessible variables for the sake of convenience.
  • EventData (internal): Type declaration for the payload being sent from the tracker. Marked as internal as it should not be relevant to the average developer, but a nice reference to have nonetheless.

We also support dark mode. You'll find it under "Settings" in the sidebar.

Good to know before implementing

NO PERSONAL INFORMATION SHOULD BE SENT!

With that over and done with, it's worth being aware that the goal of web-behavior-tracker is to replace a lot of the tracking that is done/has been done through sb1-adobe-analytics. Team Nettsider has throughout the years used a lot of specific tracking that the average digital bank team does not need to deal with. This is functionality that is also continued in web-behavior-tracker, but it can be perceived as a bit of mental overhead in this package.

As a starting point the digital bank teams only need to deal with the following tracking calls:

  • page()
  • buttonClicked()
  • formInitiated(), formStepStarted(), formStepCompleted(), formCompleted(), formErred() and formCancelled().

In addition, we have a set of naming conventions, meant to ensure data quality on our deliveries. We appreciate it if you get familiar with these.

Where is the data sent?

As a starting point you choose where you want to send the data by using the destinations array at initialization. At this time we support ADOBE, HUMIO and DATAVAREHUS. Ideally we want data to be sent to all of those, but there are numerous reasons not to do so (for instance unfinished DPIAs and complexities regarding data processor agreements within the organization). Ask when in doubt.

When data is sent to Humio, they are put in one of these repos:

  • webadferdsdata-prod
  • webadferdsdata-qa
  • webadferdsdata-test

Getting started

Installation

 
npm install @sb1/web-behavior-tracker

When testing with jest, it may be necessary to mock @adobe/alloy. This is done by adding the following to jest.setup.ts, alternatively before you do function call towards web-behavior-tracker from the tests.

jest.mock('@adobe/alloy', () => ({ createInstance: jest.fn() }));

Initializing the tracker

The first thing you do is initialize a tracker. This can be done either by using initWebBehaviorTracker(), initGlobalTracker() or with withDigitalbankDefaults(). withDigitalbankDefaults() simplify the usage for regular digitalbank teams where URLs and cookies are common knowledge. The tracker has to be initialized with configuration values that are specific for your application. The documentation for the different parameters are here.

initWebBehaviorTracker

If you use initWebBehaviorTracker, your application is responsible for keeping control of the instance.

import { getCookie, initWebBehaviorTracker, extractBankFromUri } from '@sb1/web-behavior-tracker';
import sb1OptinChecker from '@sparebank1/optin-checker'; // Safe to use from version ^2.1.0

const tracker = initWebBehaviorTracker(() => ({
    user: {
      anonymousId: getCookie('DECLAREDID'),
      language: 'nb',
      sessionId: getCookie('DSESSIONID'),
      optins: {
        statistics: sb1OptinChecker.hasOptedIn(OPTIN_TYPES.STATISTIKK),
        functional: sb1OptinChecker.hasOptedIn(OPTIN_TYPES.FUNKSJONELL),
        marketing: sb1OptinChecker.hasOptedIn(OPTIN_TYPES.MARKEDSFORING),
      },
    },
    adobe: {
      env: 'dev',
      product: { category: 'kort', name: 'refinansiering' },
      identityCookiesEnabled: true,
      businessUnit: 'SpareBank 1 Kreditt',
      subBrand: 'kreditt',
    },
    application: {
      name: 'kundefront-pm-credit-card-refinancing',
      version: '3.0.0',
      context: appContext,
    },
    destinations: ['HUMIO', 'ADOBE'],
    bank: 'none',
}))

initGlobalTracker

If you don't want to keep control of the tracker instance yourself, you can use initGlobalTracker. This function saves the instance on the window object, and is accessible by calling getGlobalTracker() afterwards. A gotcha is cases where multiple applications use initGlobalTracker on the same page. As of today we don't have any mechanisms to prevent this, therefore you should be aware that there may already exist such a tracker when you start implementing. The consequence of initializing the global tracker multiple times will be that every event from the current page will use the input configuration last applied.

import {getCookie, initGlobalTracker, extractBankFromUri, extractBankSubBrandFromUri} from '@sb1/web-behavior-tracker';
import sb1OptinChecker from '@sparebank1/optin-checker'; // Safe to use from version ^2.1.0

initGlobalTracker(() => ({
    user: {
        anonymousId: getCookie('DECLAREDID'),
        language: getLanguage(),
        sessionId: getCookie('DSESSIONID'),
        optins: {
            statistics: sb1OptinChecker.hasOptedIn(OPTIN_TYPES.STATISTIKK),
            functional: sb1OptinChecker.hasOptedIn(OPTIN_TYPES.FUNKSJONELL),
            marketing: sb1OptinChecker.hasOptedIn(OPTIN_TYPES.MARKEDSFORING),
        },
    },
    adobe: {
        // Use 'dev' here until further notice. This parameter is here to help during a transition period where we route Adobe Analytics data to the production suite. You'll hear from us when 'prod' can and should be used.
        env: 'dev',
        product: {category: 'kort', name: 'refinansiering'},
        identityCookiesEnabled: true,
        businessUnit: 'SpareBank 1',
        subBrand: extractBankSubBrandFromUri(),
    },
    application: {
        name: 'kundefront-pm-din-app',
        version: '1.0.0',
        context: appContext,
    },
    destinations: ['HUMIO', 'ADOBE', 'DATAVAREHUS'],
    bank: extractBankFromUri(),
}))

// Anywhere in the code, the tracker can be fetched
import { getGlobalTracker } from '@sb1/web-behavior-tracker';

const tracker = getGlobalTracker();

Example usage

page

After initializing a tracker, you should use the method page to register a page load. Here we can override url, referrer og title in cases where that is needed. An example of this is if the page title contains for example a customer's account number or other identifiers.

const { page } = getTracker();

// When a page is loaded
page();

page({
  url: 'https://www.sparebank1.no/nb/bank/privat.html',  
  referrer: 'https://www.sparebank1.no/nb/bank/privat/sparing.html', 
  title: 'SpareBank 1',  
});

buttonClicked

When the customer clicks links or buttons you can use buttonClicked.

const tracker = getTracker();

Generating events

Events are sent to /api/tracking/webbehavior. This API expects events on a strongly typed format. Details around models can be found in src/types.

Release new package

New packages are released as part of the build process on both master and feature-branches. Before merging your PR, run set-version.sh YOUR.VERSION.NUMBER in order to set the version in all the correct places (src/mapContext.ts, package.json and CHANGELOG.md). Remember to document your changes in the CHANGELOG. Once your change has been approved, merge the PR and a new package is published to nexus by Jenkins.

Generating and deploying documentation (Shifter)

Building and deploying the documentation is done automatically through buildCI.sh. If you want to test it locally see below:

Generate documentation:

  1. Run: npm run doc or ./doc/buildDocumentation.sh
  2. Generated documentation can be found in /docs

Deploy documentation to OpenShift:

  1. Log in: bob openshift login
  2. Run: npm run doc:deploy or deployDocumentation.sh

Note:

  • Path to index: /web-behavior-tracker/docs/index.html
  • The deploy script uses the files in the static folder. Make sure they are present.
  • The deploy-script takes the current branch into account when creating the container.
