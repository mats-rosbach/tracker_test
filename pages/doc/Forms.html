Form Tracking

In order to track a customer through a form, we have exposed the following methods:

formInitiated

formInitiated is called when the form has finished rendering on a page. In other words, the event should be triggered regardless of whether the customer has actually interacted with the form. The rule of thumb is: If you have a simple page containing a form, the number of formInitiated-events should be equal to the number of page-events for that specific page. formStepName is optional, and if no value is given it defaults to initiated. When the event is sent to Adobe it will also default to initiated regardless of the given value.

formStepStarted

formStepStarted should be called on the first interaction the user does in a form for a specific form step. A form step is defined as a set of inputfields that are rendered together on the same page, before loading the next step of the form. If your form contains one form step, formInitiated() should be called when the form has rendered, formStepStarted() on the first interaction, then followed by a formCompleted() after submit. Similarly, if your form contains two or more steps, formInitiated() should be called when the form has rendered, formStepStarted() on the first interaction of the first step. When the user navigates to the next step of the form, a new call to formStepStarted() should be performed once the user interacts with an input. This goes on all the way until the form is submitted, in which case formCompleted() should be called.

We've made a couple of examples further down the page.

formStepCompleted

formStepCompleted is an optional event for when a form step is completed. The event is only intended for logging purposes and exists to make it easier to keep track of user behavior. Adobe does not need this information, and it will be filtered.

formCompleted

formCompleted should be called after submitting the form, once you know the outcome of the form submission. If an error occurs, refer to formErred described below. If the form was submitted successfully, use one of the possible submitStatus -values to indicate the outcome. formStepName is optional, and if no value is given it defaults to completed. When the event is sent to Adobe it will also default to completed regardless of the given value.

formErred

formErred should be used if an error occurs at some point when filling out the form. There is one exception to the rule, which is validation errors. Validation errors which does not prevent the customer from resubmitting the form should not result in a formErred()-event.

formCanceled

formCanceled should be used if the form offers a dedicated "Cancel"-button which is used by the customer.

Properties

Form Name (formName)

A form name is a unique name per form. It consists of three components: section, product and action, separated by an underscore (_)

  • section - The section the form belongs to. For instance: "kontoer", "sparing", "kort", "forsikring", "lån".
  • product - (optional) - Which product does the form belong to. For instance: "brukskonto", "sparekonto", " aksjesparekonto".
  • action - Which action is performed when submitting the form. For instance: "søknad", "opprett", "sperr-kort"

If one of the components contains several words, use a hyphen (-) to separate the two. For instance: sparing_aksjesparekonto_opprett-spareavtale. The product-part is optional. If the form is not tied to a specific product, use section_action.

Form Step Name (formStepName)

formStepName should, obviously, be unique within a single formName. It is required for all form events, even though your form only contains a single step. The reasoning behind this is that we want to avoid fragmentation of the data in Adobe Analytics. You could simply default to "step_0" if it's not applicable for your form.

Form ID (formId)

In Adobe Analytics, a combination of formName and cookies is used to distinguish between a single customers journey through the form. When setting up a similar dashboard in humio, we don't have those same mechanisms. Thus, we have introduced formId. Think of it as a correlationId for humio, which will tie all form*-events for a single form submission together.

  • formId should be generated as a uuid (import { uuid } from '@sb1/web-behavior-tracker' should do the trick).
  • formId should remain the same for a single user through a single form process. If the user restarts the process, a new formId should be created.

Examples

Create account

The Create Account-form contains one input-parameter, a submit-button and a cancel-button. It does not contain any other formSteps, and we end up with the following tracking.

const tracker = getTracker();
const formId = uuid();
const formStepName = "opprett-brukskonto"
const formName = 'kontoer_brukskonto_opprett-brukskonto';

// When the current page is rendered. 
tracker.formInitiated({ formId, formName, formStepName });

// When the customer starts typing in the input field.
tracker.formStepStarted( { formId, formName, formStepName })

// When the customer has pressed "Opprett konto"/"Create Account". Typically this would be the "Next step"/"Neste steg" button.
tracker.formStepCompleted( { formId, formName, formStepName })

// Called after the customer has pressed "Opprett konto"/"Create Account" and the server has responded with an outcome
tracker.formCompleted({ formId, formName, formStepName, submitStatus });

// Customer clicked "Avbryt"/"Cancel"
tracker.formCanceled({ formId, formName, formStepName });
