Introduction

The web behavior tracker is a library that makes it easy for teams to track and send web behavior events to SpareBank1's data platform. The best way to read this documentation is to head over to our documentation page. (temp URL).

If you're new to Signal and tracking, we recommend you read through the pages listed below:

  • Overview
  • Privacy
  • Cookies
  • Data quality
  • Naming conventions
  • Migrating from Adobe SDK
  • Dictionary

Do you need any help or assistance? Send us a message in slack

Getting started

Installation

 
npm install @sb1/web-behavior-tracker

Initializing the tracker

Initializing the tracker is done by either

  • initWebBehaviorTracker
  • initGlobalTracker

In addition, we have two helper functions: withDigitalBankDefaults and withInternalDefaults which can simplify initialization depending on your context. Documentation for every initialization param is found here.

initWebBehaviorTracker

An implementation of initWebBehaviorTracker may look as follows:

import { getCookie, initWebBehaviorTracker, extractBankFromUri } from '@sb1/web-behavior-tracker';
import sb1OptinChecker from '@sparebank1/optin-checker';

const tracker = initWebBehaviorTracker(() => ({
  user: {
    anonymousId: getCookie('DECLAREDID'),
    language: 'nb',
    sessionId: getCookie('DSESSIONID'),
    optins: {
      statistics: sb1OptinChecker.hasOptedIn(OPTIN_TYPES.STATISTIKK),
      functional: sb1OptinChecker.hasOptedIn(OPTIN_TYPES.FUNKSJONELL),
      marketing: sb1OptinChecker.hasOptedIn(OPTIN_TYPES.MARKEDSFORING),
    },
  },
  adobe: {
    adobeEnv: 'dev',
    identityCookiesEnabled: true,
    businessUnit: 'SpareBank 1 Kreditt',
    subBrand: 'kreditt',
  },
  application: {
    name: 'kundefront-pm-credit-card-refinancing',
    version: '3.0.0',
    context: appContext,
  },
  destinations: ['HUMIO', 'ADOBE'],
  bank: 'smn',
  product: { category: 'kort', name: 'refinansiering' },
}))

initGlobalTracker

If you don't want to keep control of the tracker instance yourself, you can use initGlobalTracker. This function saves the instance on the window object, and is accessible by calling getGlobalTracker() afterwards. A gotcha is cases where multiple applications use initGlobalTracker on the same page. As of today we don't have any mechanisms to prevent this, therefore you should be aware that there may already exist such a tracker when you start implementing. The consequence of initializing the global tracker multiple times will be that every event from the current page will use the input configuration last applied. Initializing a global tracker may look as follows:

import { getCookie, initGlobalTracker, extractBankFromUri, extractBankSubBrandFromUri } from '@sb1/web-behavior-tracker';
import sb1OptinChecker from '@sparebank1/optin-checker'; // Safe to use from version ^2.1.0

initGlobalTracker(() => ({
  user: {
    anonymousId: getCookie('DECLAREDID'),
    language: getLanguage(),
    sessionId: getCookie('DSESSIONID'),
    optins: {
      statistics: sb1OptinChecker.hasOptedIn(OPTIN_TYPES.STATISTIKK),
      functional: sb1OptinChecker.hasOptedIn(OPTIN_TYPES.FUNKSJONELL),
      marketing: sb1OptinChecker.hasOptedIn(OPTIN_TYPES.MARKEDSFORING),
    },
  },
  adobe: {
    // Use 'dev' here until further notice. This parameter is here to help during a transition period where we route Adobe Analytics data to the production suite. You'll hear from us when 'prod' can and should be used.
    adobeEnv: 'dev',
    identityCookiesEnabled: true,
    businessUnit: 'SpareBank 1',
    subBrand: extractBankSubBrandFromUri(),
  },
  application: {
    name: 'kundefront-pm-din-app',
    version: '1.0.0',
    context: appContext,
  },
  destinations: ['HUMIO', 'ADOBE', 'DATAVAREHUS'],
  bank: extractBankFromUri(),
  product: { category: 'kort', name: 'refinansiering' },
}))

// Anywhere in the code, the tracker can be fetched
import { getGlobalTracker } from '@sb1/web-behavior-tracker';

const tracker = getGlobalTracker();

Defaults

We have two sets of defaults:

  • withDigitalBankDefaults
  • withInternalDefaults

If you have a "standard" digitalbankapp, using withDigitalBankDefaults greatly simplifies the initialization:

const myTracker = initWebBehaviorTracker(() => withDigitalBankDefaults({
  application: {
    name: "kundefront-pm-sparing",
    version: "1.3.0",
  },
  language: LanguageEnum.nb,
  adobeEnv: AdobeEnvEnum.Dev
}));

If you have an internal application, not used by customers, you can use withInternalDefaults:

const myTracker = initGlobalTracker(() => withInternalDefaults({
  application: {
    name: 'kontorfront-developer-portal',
  },
  language: LanguageEnum.en,
  backendEnvironment: 'prod'
}));

Example usage

The tracker contains a lot of functionality, but most teams only need to implement a handful of events:

  • page
  • buttonClicked
  • Forms

For a complete list of events, check out the WebBehaviorTracker interface

page

You can use page to register a pageLoaded-event:

const { page } = getTracker();

// page() without overrides
page();

// Page() with three attributes overridden.  For instance if your URL contains sensitive information.
page({
  url: 'https://www.sparebank1.no/nb/bank/privat.html',  
  referrer: 'https://www.sparebank1.no/nb/bank/privat/sparing.html', 
  title: 'SpareBank 1',  
});

buttonClicked

When the customer clicks links or buttons you can use buttonClicked.

const tracker = getTracker();

tracker.buttonClicked({
    text: 'Faste overføringer',
    section: 'betaling',
    product: 'overforinger-egne-konti',
    action: 'se-faste-overforinger',
    linkType: 'navigation',
});

Generating events

Events are sent to /api/tracking/webbehavior. This API expects events on a strongly typed format. Details around models can be found in src/types.

How to test
  • test app: https://stash.intern.sparebank1.no/projects/AN/repos/web-behavior-tracker-test/browse
  • can be run locally to see how the events are working in payload

Debugging in browser

As the request payload is lost when navigating to another page, it is possible to set document.cookie='print_signal=true;path=/' in the browser console to enable debugging.

Release new package

New packages are released as part of the build process on both master and feature-branches. Before merging your PR, run set-version.sh YOUR.VERSION.NUMBER in order to set the version in all the correct places (src/mapContext.ts, package.json and CHANGELOG.md). Remember to document your changes in the CHANGELOG. Once your change has been approved, merge the PR and a new package is published to nexus by Jenkins.

You can still do it locally::
