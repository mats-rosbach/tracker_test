12.0.0

The great refactoring part 2.

In order to keep up with the growing scale and complexity of the tracker the time is due for some house-cleaning. This is done in one batch in order to reduce strain for our users, where we don't want to expose them to breaking changes regularly.

A short summary of what has changed is that the "old" parts, such as AppState, BaseTracker, ApiTypes has been replaced. Over time the tracker has been built atop of these code block, result in a confusing hierarchy which was prone to bugs. Replacing these have the added benefit of using the intended type declarations, fixing bugs with missing data in the tracker output in the process. Files, functions, types and more has been renamed and moved around in order to alleviate unclarities and confusion. The documentation page has been upgraded, with more exhaustive details.

Changes:

  • AppState is removed. Relevant state is now retrieved from the InitParams type.
  • BaseTracker is removed. Replaced by the sendEvent function.
  • ApiTypes is removed. Its type definitions are now in a more sensible location.
  • Files, tests, functions, variables and types has been renamed to a more cohesive naming standard.
  • Types and tests has been moved to their own folder for easier navigation.
  • Prettier print width is now 100, for improved readability.
  • Bumped dependencies.
  • Upgraded documentation. The documentation page has a new layout, has more exhaustive details, and is now categorized.
  • The documentation now uses the plugin @knodes/typedoc-plugin-pages, to help us structure the page the way we want.
  • Thanks to the typedoc plugin, the documentation script has been trimmed down. No longer injects HTML and no longer duplicates documentation generation.
  • Clean up how we do our exports and trim away exports that is not intended for use.
  • Constant string values now has their respective Enums. Our Literal Union Types are now derived from these.
  • Creating mock data for tests now uses a single function, with more dynamic input handling.

  • Renaming PageOptions to Options. This is only breaking if you use the exported variable PageOptions.
  • Adds support for customData which is specifically targetting Humio. This makes team specific problems easier to solve. You can for instance pass "customData": {"abVariant": "A"} as thisis functionality web-behavior-tracker does not " officially" support yet. The different teams are responsible for not sending personal information. customData is passed through the Options-object (hence the renaming)
  • Adds withDigitalBankDefaults, which is an overload of digitalbankDefaults. Accepts an DigitalbankOptions-object instead of using several unnamed and often confusing parameters.

Usage

import { initWebBehaviorTracker } from "@sb1/web-behavior-tracker";
import { withDigitalbankDefaults } from "@sb1/web-behavior-tracker";
import { digitalbankDefaults } from "./initParams";

// customData will be passed with each request
const tracker = initWebBehaviorTracker(() => withDigitalbankDefaults({
  application: {
    name: "kundefront-pm-sparing",
    version: "1.3.0",
    context: "innlogget-mobilbank"
  },
  language: 'nb',
  adobeEnv: 'dev',
  customData: {
    abGroup: 'A'
  }
}));

// Or could be passed within a single request
const myTracker = initWebBehaviorTracker(() => withDigitalbankDefaults({
  application: {
    name: "kundefront-pm-sparing",
    version: "1.3.0",
    context: "innlogget-mobilbank"
  },
  language: 'nb',
  adobeEnv: 'dev'
}));

myTracker.page({ customData: { abGroup: 'A' } })

// ... And one more thing. The two approaches can be used together. 

// Finally, if you prefer to stick with `digitalbankDefaults` over `withDigitalbankDefaults`,
const myThirdTracker = initWebBehaviorTracker(() => digitalbankDefaults(
  {
    name: "kundefront-pm-sparing",
    version: "1.3.0",
    context: "innlogget-mobilbank"
  }, 'nb', 'dev', { abGroup: 'A' }
));
